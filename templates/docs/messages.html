FieldRequiredNotes
fromYesMust be on a verified sending domain for your account.
toYesA single recipient address.
subjectNoDefaults to empty.
textNoPlain-text body.
htmlNoHTML body, sent as a MIME alternative alongside text.

Sending with a template

Rather than passing subject/text/html inline on every request, create a reusable template and reference it — the same pattern campaigns use for bulk sends.

Response

A successful call queues the message and returns immediately — sending happens asynchronously.

202 Accepted
{
  "id": 4821,
  "status": "queued"
}

status reflects the message's state at accept time; it doesn't update on this response as the send completes. Subscribe to webhooks for message.sent / message.failed outcomes.

Idempotency

There is no idempotency-key support on this endpoint today. If a request times out or your connection drops after the message was actually accepted, retrying it will send a second copy — there's no way to safely replay a call and be guaranteed a single send. If duplicate delivery would be a problem for you, track sent message ids client-side (or de-duplicate on a stable value in your own data, like an invoice ID) before retrying.

Errors

See the full error codes reference. The most common ones for this endpoint:

StatusCodeCause
400invalidMissing/malformed from or to.
401authentication_failedMissing or invalid API key.
403unverified_domainfrom's domain isn't verified for your account.
403permission_deniedYour plan doesn't include the API & SMTP relay — see the message field for detail.
403emailsMonthly send volume cap reached.
429—Per-minute rate limit exceeded; see Retry-After.

A note on the legacy endpoint

POST /email/send/ (the same request/response shape, header X-Api-Key only) still works and shares this exact send logic — it's kept alive for existing integrations, but new integrations should use /api/v1/messages.
